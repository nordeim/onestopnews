# 1. Executive Vision

OneStopNews is not a news aggregator.

It is a continuously updated understanding layer for current events.

The product transforms hundreds of articles into a small number of evolving Story Objects that explain:

- What happened
- Why it matters
- What changed
- What happens next

The primary user outcome is not reading more news.

The primary user outcome is becoming informed faster.

Success is measured by:

"How quickly can a user understand today's important developments?"

rather than:

"How many articles did they click?"

# Final Strategic Assessment

The most important improvement from v2.0 to v3.0 is not technical.

It is conceptual.

The platform stops treating news as content.

It starts treating news as evolving situations.

That shift creates a clearer consumer value proposition, stronger differentiation from incumbents, a more memorable product experience, and a foundation for long-term retention that article-centric aggregators struggle to achieve.

If I were taking this into actual product design next, I would move into a dedicated UX architecture phase and produce:
- Story clustering specification
- Story intelligence data model
- Mobile-first user journeys
- Detailed screen blueprints
- Avant-garde "Living Briefing" design system
- MVP vs V1 feature prioritization matrix
- Technical feasibility assessment for story synthesis and clustering quality thresholds

Before implementation, I would recommend validating the Story-first model through user testing prototypes, as story comprehension quality will become the most important determinant of product success.

# Executive Summary

The next highest-value step is not implementation and not UI mockups.

The next step is validating whether the Story-first model can consistently produce high-quality understanding.

The greatest risk to OneStopNews is no longer architecture.

It is story quality.

If story clustering, story synthesis, and story evolution tracking are excellent, the product is differentiated.

If they are mediocre, the product becomes:

"AI summaries of article clusters"

which is exactly where many AI-news products fail.

Therefore, I recommend the next blueprint phase focus on:

1. Story Intelligence Architecture

# 2. The Three Visual Pillars

## Pillar 1 — Ink Discipline

The interface is built from typography and rules (lines), not boxes and shadows.

Hierarchy is created with:

- weight
- size
- case
- spacing

Never with:

- drop shadows
- colored cards
- gradient surfaces

---

## Pillar 2 — Wire Rhythm

Metadata everywhere uses monospace type:

```text
UPDATED 12:42 · 64 REPORTS · HIGH CONFIDENCE
```

This creates a "wire terminal" undertone — the feeling that the system is alive and continuously receiving.

---

## Pillar 3 — Calm Authority

Color is scarce and semantic.

When color appears, it means something:

- a story is breaking
- confidence changed
- a development arrived

Color is editorial ink. Never decoration.

# 3. Foundational Tokens

All tokens defined as CSS custom properties, consumed via Tailwind v4 `@theme`.

---

## 3.1 Color System

### Paper (Surfaces)

```css
--paper-0:    #FCFBF8;   /* page background — warm off-white, not pure white */
--paper-50:   #F7F5F0;   /* recessed sections */
--paper-100:  #F0EDE6;   /* dividers-as-surface, brief panels */
--paper-150:  #E7E3DA;   /* hairline borders */
```

Rationale:

Pure white (`#FFFFFF`) reads as "app."
Warm paper reads as "publication."

---

### Ink (Text)

```css
--ink-900:  #16150F;   /* headlines, why-it-matters */
--ink-700:  #33312A;   /* body text */
--ink-500:  #6B6759;   /* secondary text */
--ink-350:  #98937F;   /* metadata, timestamps */
```

Contrast verification:

```text
ink-900 on paper-0  →  ~15.8:1   (AAA)
ink-700 on paper-0  →  ~10.9:1   (AAA)
ink-500 on paper-0  →   ~5.4:1   (AA large, AA normal)
ink-350 on paper-0  →   ~3.2:1   (metadata only, never body)
```

Rule:

`--ink-350` may only be used at monospace metadata sizes with adjacent redundant signals (never sole information carrier).

---

### Signal (Semantic Accents)

```css
--signal-breaking:   #B3261E;   /* breaking / escalation — deep editorial red */
--signal-update:     #1A5BB8;   /* new development — wire blue */
--signal-confirmed:  #2E6B4F;   /* high confidence / resolution — ledger green */
--signal-developing: #9A6B15;   /* developing / caution — amber ochre */
```

Usage law:

```text
Signal colors appear ONLY as:

• 2px left rules on story modules
• Monospace status labels
• Timeline event markers

Never as:

• Backgrounds
• Buttons
• Large fills
```

---

### Dark Mode

Inverted paper, not "dark theme gray."

```css
--paper-0-dark:   #131210;   /* warm near-black */
--paper-50-dark:  #1B1A17;
--paper-100-dark: #232119;
--ink-900-dark:   #EDEAE0;
--ink-700-dark:   #C9C5B8;
--ink-500-dark:   #918D7E;
```

Dark mode should feel like:

```text
A briefing read at night
```

not

```text
Developer tools
```

## 3.2 Typography System

### Families

```css
--font-display: "Newsreader", Georgia, serif;
--font-ui:      "Space Grotesk", system-ui, sans-serif;
--font-wire:    "JetBrains Mono", ui-monospace, monospace;
```

Loading strategy:

- `next/font` with subset latin
- `font-display: swap` is forbidden for `--font-display` on Story headlines — use `optional` + size-adjusted fallback metrics to prevent layout shift on the most important element

---

### Type Scale

A deliberate non-linear scale — large jumps create editorial drama:

```css
--text-wire:     0.6875rem;  /* 11px — monospace metadata */
--text-caption:  0.8125rem;  /* 13px — labels, facts */
--text-body:     1.0625rem;  /* 17px — reading text (not 16) */
--text-lede:     1.25rem;    /* 20px — why-it-matters body */
--text-title-s:  1.5rem;     /* 24px — secondary story headlines */
--text-title-m:  2rem;       /* 32px — feed story headlines */
--text-title-l:  3rem;       /* 48px — story page headline */
--text-display:  4.25rem;    /* 68px — daily brief masthead, desktop only */
```

Note the gaps. There is no "1.125rem". Intermediate sizes are forbidden — they erode hierarchy.

---

### Line Heights

```css
--leading-display: 1.04;   /* headlines — tight, confident */
--leading-title:   1.12;
--leading-lede:    1.4;
--leading-body:    1.62;   /* generous reading rhythm */
--leading-wire:    1.0;    /* monospace metadata — single line */
```

---

### Typographic Roles (Locked Pairings)

| Role | Family | Size | Weight | Case |
|---|---|---|---|---|
| Story headline | Newsreader | title-l | 500 | Sentence |
| Feed headline | Newsreader | title-m | 500 | Sentence |
| Why This Matters label | Space Grotesk | caption | 700 | UPPERCASE, +0.08em tracking |
| Why This Matters body | Newsreader | lede | 400 italic-optional | Sentence |
| Body / Summary | Space Grotesk | body | 400 | Sentence |
| Section labels | Space Grotesk | caption | 700 | UPPERCASE |
| Metadata / status | JetBrains Mono | wire | 500 | UPPERCASE |
| Timeline dates | JetBrains Mono | wire | 700 | UPPERCASE |

These pairings are non-negotiable. They ARE the brand.

---

## 3.3 Spacing System

Base unit: `4px`. But spacing is used asymmetrically — editorial pages breathe vertically, stay disciplined horizontally.

```css
--space-1:  0.25rem;
--space-2:  0.5rem;
--space-3:  0.75rem;
--space-4:  1rem;
--space-6:  1.5rem;
--space-8:  2rem;
--space-12: 3rem;
--space-16: 4rem;
--space-24: 6rem;
```

Vertical rhythm law:

```text
Between modules within a story:    space-8
Between stories in the feed:       space-16
Above a section label:             space-12
Below a section label:             space-3   (label hugs its content)
```

The asymmetry (12 above, 3 below) visually binds labels to content — a print convention almost no app follows.

---

## 3.4 Rules & Dividers (The Box Replacement)

Since cards are forbidden, structure comes from rules:

```css
--rule-hair:   1px solid var(--paper-150);     /* default separators */
--rule-story:  2px solid var(--ink-900);       /* story boundaries */
--rule-signal: 2px solid var(--signal-*);      /* left-edge status rules */
```

The Story Boundary:

Every story in the feed is separated by a full-width `2px` ink rule. This is the system's most recognizable structural signature:

```text
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

STORY

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

---

## 3.5 Radius & Elevation

```css
--radius-none: 0;        /* default — almost everything */
--radius-xs:   2px;      /* interactive elements only (buttons, inputs) */
```

Shadows:

```css
/* Permitted ONLY for: mobile bottom sheet, command palette */
--shadow-sheet: 0 -1px 0 var(--paper-150), 0 -24px 48px -12px rgb(22 21 15 / 0.18);
```

Everything else is flat. Depth comes from typography, not z-axis.

# 5. Motion System

## Philosophy

```text
Motion communicates change in information.
Nothing else moves.
```

---

## Permitted Motion

| Event | Motion | Duration |
|---|---|---|
| New update arrives in Wire Rail | Single fade-in + 2px signal rule draws downward | 240ms |
| Story page navigation | View Transition: headline morphs from feed to story page | 320ms |
| "Since your last visit" counts | Number ticks up once on mount | 400ms |
| Timeline reveal on scroll | Events fade in sequentially, 40ms stagger | 180ms each |

---

## Forbidden Motion

```text
❌ Parallax
❌ Hover lift/scale on stories
❌ Skeleton shimmer (use static placeholder text instead)
❌ Looping/ambient animation
❌ Scroll-jacking
```

---

## Reduced Motion

`prefers-reduced-motion: reduce` →

All transitions become instant opacity swaps. View Transitions disabled. Number ticks render final value immediately.

# 3. Data Contracts (Day One Artifact)

```typescript
// domain/story/types.ts
import { z } from "zod";

export const confidenceLevelSchema = z.enum([
  "high",
  "moderate",
  "developing",
]);

export const storyStatusSchema = z.enum([
  "emerging",
  "active",
  "mature",
  "archived",
]);

export const timelineEventSchema = z.object({
  id: z.string(),
  title: z.string(),
  summary: z.string(),
  occurredAt: z.coerce.date(),
  isLatest: z.boolean(),
});

export const storySourceSchema = z.object({
  name: z.string(),
  articleCount: z.number().int().positive(),
});

export const storyUpdateSchema = z.object({
  id: z.string(),
  storyId: z.string(),
  title: z.string(),
  summary: z.string(),
  createdAt: z.coerce.date(),
});

export const storySchema = z.object({
  id: z.string(),
  slug: z.string(),
  title: z.string(),
  category: z.string(),
  status: storyStatusSchema,
  isBreaking: z.boolean(),
  confidence: confidenceLevelSchema,
  whyItMatters: z.string().max(600),
  latestUpdate: z.string().max(400),
  executiveSummary: z.string(),
  keyFacts: z.array(z.string()).max(10),
  timeline: z.array(timelineEventSchema),
  sources: z.array(storySourceSchema),
  articleCount: z.number().int(),
  sourceCount: z.number().int(),
  firstSeenAt: z.coerce.date(),
  updatedAt: z.coerce.date(),
});

export type Story = z.infer<typeof storySchema>;
export type StoryUpdate = z.infer<typeof storyUpdateSchema>;
export type TimelineEvent = z.infer<typeof timelineEventSchema>;
export type ConfidenceLevel = z.infer<typeof confidenceLevelSchema>;
```


```typescript

}
```

Note `parse` — fixtures self-validate against the contract. A drifting fixture fails at build, not in review.

```typescript
  id: z.string(),
  storyId: z.string(),
  title: z.string(),
  summary: z.string(),
  createdAt: z.coerce.date(),
});

export const storySchema = z.object({
  id: z.string(),
  slug: z.string(),

  category: z.string(),
  status: storyStatusSchema,
  isBreaking: z.boolean(),
  confidence: confidenceLevelSchema,

  latestUpdate: z.string().max(400),
  executiveSummary: z.string(),
  keyFacts: z.array(z.string()).max(10),
  timeline: z.array(timelineEventSchema),
  sources: z.array(storySourceSchema),
  articleCount: z.number().int(),
  sourceCount: z.number().int(),
  firstSeenAt: z.coerce.date(),
  updatedAt: z.coerce.date(),
});

export type Story = z.infer<typeof storySchema>;
export type StoryUpdate = z.infer<typeof storyUpdateSchema>;
export type TimelineEvent = z.infer<typeof timelineEventSchema>;
```

# Part 4 — Voice Rules as Tested Logic

```typescript
// src/shared/lib/time.ts
const MINUTE_MS = 60_000;
const HOUR_MS = 60 * MINUTE_MS;

const MONTHS = [
  "JAN", "FEB", "MAR", "APR", "MAY", "JUN",
  "JUL", "AUG", "SEP", "OCT", "NOV", "DEC",
] as const;

/**
 * Voice rules (Design Spec §7):
 * - under 1 minute  → "JUST NOW"
 * - under 1 hour    → "12M AGO"
 * - 1 hour or older → "14 JUN, 09:12"
 * Future timestamps clamp to "JUST NOW" (clock-skew tolerance).
 */
export function formatStoryTime(date: Date, now: Date = new Date()) {
  const diff = now.getTime() - date.getTime();

  if (diff < MINUTE_MS) return "JUST NOW";
  if (diff < HOUR_MS) return `${Math.floor(diff / MINUTE_MS)}M AGO`;

  const day = String(date.getDate()).padStart(2, "0");
  const month = MONTHS[date.getMonth()];
  const hours = String(date.getHours()).padStart(2, "0");
  const minutes = String(date.getMinutes()).padStart(2, "0");
  return `${day} ${month}, ${hours}:${minutes}`;
}

/** Voice rule: counts are precise. "64 REPORTS", never "60+". */
export function formatReportCount(count: number) {
  return `${count} ${count === 1 ? "REPORT" : "REPORTS"}`;
}
```

```typescript
describe("formatStoryTime", () => {
  it("returns JUST NOW under one minute", () => {
    expect(formatStoryTime(new Date("2025-06-14T12:41:30"), NOW)).toBe("JUST NOW");
  });

  it("returns relative minutes under one hour", () => {
    expect(formatStoryTime(new Date("2025-06-14T12:30:00"), NOW)).toBe("12M AGO");
  });

  it("returns relative at 59 minutes", () => {
    expect(formatStoryTime(new Date("2025-06-14T11:43:00"), NOW)).toBe("59M AGO");
  });

  it("switches to absolute at exactly one hour", () => {
    expect(formatStoryTime(new Date("2025-06-14T11:42:00"), NOW)).toBe("14 JUN, 11:42");
  });

  it("pads day, hour, and minute", () => {
    expect(formatStoryTime(new Date("2025-06-04T09:05:00"), NOW)).toBe("04 JUN, 09:05");
  });

  it("clamps future timestamps to JUST NOW", () => {
    expect(formatStoryTime(new Date("2025-06-14T12:43:00"), NOW)).toBe("JUST NOW");
  });
```
